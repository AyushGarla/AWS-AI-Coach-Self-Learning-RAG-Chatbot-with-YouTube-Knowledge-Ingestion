Amazon Web Services (AWS) is a comprehensive and widely adopted cloud computing platform developed by Amazon. AWS provides a broad set of infrastructure services, such as computing power, storage options, and networking capabilities, along with a vast range of higher-level services including databases, analytics, machine learning, developer tools, security, and enterprise applications.

## Key Features of AWS

- **Global Infrastructure**: AWS operates data centers worldwide, organized into Regions and Availability Zones, allowing users to deploy applications close to end-users for improved latency and redundancy.
- **Scalability**: AWS enables users to automatically scale resources up or down based on demand, ensuring cost-effective operation and responsiveness.
- **Pay-as-You-Go Pricing**: Customers only pay for the resources they use, with no upfront costs or long-term contracts required.
- **Security**: AWS offers robust security features, including encryption, compliance certifications, network protections, identity management, and monitoring tools.
- **Wide Range of Services**: AWS offers over 200 services, from core infrastructure like EC2 (virtual servers) and S3 (object storage), to advanced services for artificial intelligence, the Internet of Things (IoT), mobile, and more.

## Common Use Cases

- **Application Hosting**: Launch and manage scalable web and mobile applications.
- **Data Storage and Archiving**: Store large amounts of structured and unstructured data efficiently and securely.
- **Big Data Analytics**: Run complex analytics workloads on massive data volumes using tools like Amazon Redshift and AWS Glue.
- **Machine Learning and AI**: Build, train, and deploy machine learning models using managed services, such as Amazon SageMaker.
- **Backup and Disaster Recovery**: Implement redundant and reliable backup solutions for data and applications.

## Benefits of AWS

- **Reliability**: AWS infrastructure is designed for high availability and fault tolerance.
- **Flexibility**: Easily choose the right mix of services and resources to suit specific workloads and business requirements.
- **Innovation**: AWS regularly releases new services and features, enabling customers to adopt the latest technologies quickly.
- **Developer Ecosystem**: Access to APIs, SDKs, documentation, and support communities for rapid development.

## AWS Account Basics

- To use AWS, users sign up for an AWS account, which provides access to the AWS Management Console, CLI, and programmatic interfaces.
- AWS accounts can be managed individually or consolidated with AWS Organizations for centralized billing and policy management.

## Learning Resources

- **AWS Documentation:** Official guides and reference material for every service.
- **AWS Training and Certification:** Courses, hands-on labs, and certifications for cloud professionals.
- **AWS Support:** Tiered support options, online forums, and troubleshooting tools.

AWS is foundational for modern cloud development and offers the tools needed for virtually any workload, from startups to enterprises.

AWS Global Infrastructure refers to the worldwide network of facilities, hardware, and software that supports all AWS services. This infrastructure is purpose-built to provide scalability, reliability, security, and flexibility for customers globally.

## Regions

- A Region is a geographic area that contains multiple, isolated locations known as Availability Zones.
- Regions are designed to be fully independent to provide fault tolerance and stability.
- Users can select a Region to deploy resources close to their end-users or to meet compliance requirements.
- Each Region is physically separated and isolated from every other Region.

## Availability Zones

- An Availability Zone (AZ) is one or more discrete data centers within a Region, each with redundant power, networking, and connectivity.
- AZs are physically separated but close enough to support synchronous data replication.
- Deploying resources across multiple AZs increases reliability and fault tolerance.
- Most Regions have two or more AZs.

## Edge Locations

- Edge Locations are data centers that manage requests for AWS services close to users, mainly for content delivery via Amazon CloudFront and AWS Global Accelerator.
- They cache content and accelerate delivery of web applications, videos, and APIs.
- There are hundreds of Edge Locations worldwide for low-latency connectivity.

## Local Zones

- Local Zones are extensions of AWS Regions that place compute, storage, database, and other select services closer to large population and industry centers.
- They allow latency-sensitive applications to run closer to end users, reducing delay for critical workloads.
- Local Zones are suitable for real-time gaming, multimedia, and hybrid cloud deployments that require local access.

 AWS Identity and Access Management (IAM) enables secure access control for AWS resources. IAM lets you manage users, groups, roles, and permissions centrally within your AWS account to ensure only authorized access to services and data.

## IAM Users, Groups, and Roles

- **Users**: Individual identities with credentials (username, password, and optional access keys) for accessing AWS resources. Each user can be assigned specific permissions via policies.
- **Groups**: Collections of users that simplify permission management. Permissions assigned to a group apply to all its members.
- **Roles**: Identities that can be assumed by trusted users, AWS services, or external accounts. Roles provide temporary security credentials and follow the principle of least privilege, often used for delegation or cross-account access.

## IAM Policies

- Policies are documents written in JSON that define permissions for users, groups, and roles.
- Policies specify ‚ÄúAllow‚Äù or ‚ÄúDeny‚Äù actions on AWS resources, indicating who can perform which operations and under what conditions.
- Policies can be AWS-managed or customer-managed, tailored for organizational requirements.

## Multi-Factor Authentication (MFA)

- MFA adds an extra layer of security to AWS accounts by requiring an additional form of authentication beyond username and password.
- Supports virtual (software-based) devices, hardware key fobs, and biometric devices.
- Enabling MFA is strongly recommended for AWS root accounts and privileged users to reduce the risk of unauthorized access.

## IAM Best Practices

- Follow the principle of least privilege: grant only the minimum permissions required for tasks.
- Enable MFA for all users, especially the root user.
- Use IAM roles for applications and EC2 instances instead of long-term access keys.
- Regularly review and audit permissions, removing unnecessary users and credentials.
- Rotate credentials and access keys frequently, and avoid embedding them in code or repositories.
- Apply policies at the group or role level instead of directly to users when possible, to simplify management.

 Amazon Elastic Compute Cloud (Amazon EC2) is a web service that provides resizable compute capacity in the cloud, enabling users to launch virtual servers (instances) for scalable and secure application hosting.

## EC2 Instance Types

- Amazon EC2 offers a variety of instance types optimized for different workloads (compute, memory, storage, and GPU).
- Popular categories include:
  - General Purpose (e.g., t3, m6i): Balanced compute, memory, and networking.
  - Compute Optimized (e.g., c7g): Best for high-performance processing.
  - Memory Optimized (e.g., r6i): Ideal for in-memory databases and analytics.
  - Storage Optimized (e.g., i4i): For workloads requiring high storage throughput.
  - Accelerated Computing (e.g., p4, g6): Suitable for machine learning and high-performance GPU tasks.

## Pricing Models

- **On-Demand Instances**: Pay for compute capacity by the hour or second without long-term commitments; ideal for unpredictable workloads.
- **Reserved Instances**: Commit to one or three years for a discounted rate; best for steady-state usage.
- **Spot Instances**: Purchase spare capacity at reduced prices, suitable for flexible, interruption-tolerant tasks.
- Additional options such as Savings Plans allow further customization and cost control.

## Security Groups

- Security groups act as virtual firewalls that control inbound and outbound traffic to EC2 instances.
- You can specify allowed IP ranges, ports, and protocols; security groups are stateful (responses to allowed inbound traffic are automatically permitted).
- Security groups provide an essential layer of defense for EC2 resources.

## Key Pairs

- Key pairs are used for secure SSH access to EC2 instances running Linux or to decrypt administrator passwords for Windows instances.
- AWS stores the public key, while the user retains the private key, which is required for authentication.
- It is critical to safeguard private keys and avoid sharing them or storing them insecurely.

## Amazon Elastic Block Store (EBS)

- Amazon EBS offers persistent block storage volumes for EC2 instances.
- EBS volumes can be attached, detached, and resized independently of instances, providing flexible and durable storage.
- Snapshot functionality allows you to back up EBS volumes for disaster recovery, migration, or cloning.
- EBS provides several volume types optimized for performance, throughput, or cost.

 Amazon Simple Storage Service (Amazon S3) provides scalable object storage for a wide range of use cases, including backup, archival, analytics, and hosting static content.[3][7]

## Buckets & Objects

- **Buckets** are containers for storing objects (files) in S3. Each bucket must have a globally unique name and is created in a specific AWS Region.
- **Objects** consist of data, metadata, a key (unique identifier), and a version ID if versioning is enabled. There is no limit to the number of objects you can store in a bucket, and each object can be up to 5 TB.[6][7][3]

## Storage Classes

- S3 offers multiple storage classes to optimize cost and performance:
  - **S3 Standard**: For frequently accessed data.
  - **S3 Intelligent-Tiering**: Automatically moves data between access tiers based on usage.
  - **S3 Standard-IA (Infrequent Access)**: For less frequently accessed but critical data.
  - **S3 One Zone-IA**: Data is stored in a single AZ, suitable for recreatable, infrequent data.
  - **S3 Glacier Instant Retrieval**: For archive data needing instant access.
  - **S3 Glacier Flexible Retrieval**: Long-term archival with flexible retrieval times.
  - **S3 Glacier Deep Archive**: Lowest cost, long-term archival, retrieval in hours.
  - **S3 Outposts/Express One Zone**: For on-premises or high-performance access in local zones.[1][2][5]

## Lifecycle Policies

- Lifecycle policies automate transitions of objects between storage classes, enable expiration or deletion, and reduce costs by managing data retention.
- Configure policies at the bucket or prefix level to move or delete objects on a schedule based on rules you define.[2][3]

## Versioning

- Versioning keeps multiple variants of an object in a bucket, protecting against accidental deletion or overwrites.
- Each object version is assigned a unique version ID, and deleting an object results in the creation of a delete marker rather than permanent removal.[7]
- Versioning can be enabled or suspended per bucket.

## Static Website Hosting

- S3 buckets can host static websites by storing HTML, CSS, images, and JavaScript files.
- Enable static website hosting in the bucket properties, designate an index and error document, and configure public access permissions.[3]
- The bucket provides an endpoint to access the hosted site.

 Amazon Virtual Private Cloud (VPC) lets you provision a logically isolated network within the AWS cloud where you launch and manage AWS resources securely.[1][3][7]

## Subnets

- Subnets divide the VPC‚Äôs IP address range into smaller segments for grouping resources.
- Each subnet resides entirely in one Availability Zone, supporting zonal isolation and high availability.
- Public subnets route traffic through an Internet Gateway, while private subnets do not allow direct internet access.[3][6]

## Route Tables

- Route tables define how network traffic is directed within your VPC.
- Each subnet must be associated with one route table, listing destination CIDR blocks and traffic targets.
- Routes can direct traffic internally (between subnets), externally (to Internet Gateway or NAT Gateway), or to other resources like VPNs.[4][3]

## Internet Gateway

- An Internet Gateway is a horizontally scaled, redundant gateway that allows resources in your VPC with public IP addresses to access the internet.
- Attach an Internet Gateway to your VPC to enable outbound and inbound internet traffic for public subnets.[7][4]

## NAT Gateway

- A NAT Gateway enables instances in a private subnet to access the internet or other AWS services, while preventing inbound access from the internet.
- NAT Gateway is highly available and scales automatically with traffic.[4][7]

## Security Groups vs Network ACLs (NACLs)

- **Security Groups** are stateful, instance-level firewalls controlling inbound/outbound traffic; changes are reflected immediately, and return traffic is automatically allowed if initiated by the instance.
- **Network ACLs (NACLs)** are stateless, subnet-level filters; they require explicit inbound and outbound rules, and apply to all resources within a subnet.
- Use security groups for granular instance protection and NACLs for network-wide controls.[6][9][4]

## VPC Peering

- VPC Peering lets you privately connect two VPCs, either in the same or different AWS accounts and regions.
- Communication between VPCs is routed using private IP addresses, without traversing the public internet.
- Peering does not support transitive routing; each connection is direct and must be configured individually.[3][6]

Amazon CloudFront is a content delivery network (CDN) service that accelerates the distribution of both static and dynamic web content by caching copies of your files at edge locations worldwide.[1][3]

## CDN Basics

- CloudFront routes user requests to the nearest edge location (data center) based on network latency, ensuring faster delivery.
- Content is cached at these edge locations so repeated requests can be served quickly without going back to the origin server.
- This reduces latency, decreases load on your origin infrastructure, and improves overall user experience for globally distributed users.[2][1]

## Caching

- When a user requests content, CloudFront first checks if the content is available in the cache at the edge location.
- If the content is cached (cache hit), it is delivered immediately.
- If not cached (cache miss), CloudFront fetches the content from the origin server, serves it to the user, and caches it for subsequent requests.
- Cache behavior settings allow control over TTL (time-to-live), what headers and cookies are forwarded, and cache invalidation lets you update content before TTL expiration.[3][6]

## Origins

- Origins are the source locations where your original content is stored.
- Common origins include Amazon S3 buckets, Amazon EC2 instances, Elastic Load Balancers, or custom HTTP servers.
- You specify one or more origins when creating a CloudFront distribution; CloudFront retrieves content from these origin servers on cache misses.
- Origin Failover allows configuration of multiple origins for high availability, automatically switching to a secondary origin if the primary is unavailable.[6][1]

 Amazon Elastic Load Balancing (ELB) offers three main types of load balancers, each designed to serve different traffic patterns and use cases.[1][3][9]

## Classic Load Balancer (CLB)

- The original ELB option, operating at both Layer 4 (transport) and Layer 7 (application).
- Distributes incoming traffic across multiple EC2 instances for basic load balancing.
- Supports HTTP, HTTPS, TCP, and SSL.
- Limited features compared to newer load balancers; AWS recommends using ALB or NLB for most scenarios.
- Suitable for legacy applications and environments, especially those using EC2-Classic.

## Application Load Balancer (ALB)

- Works at Layer 7 (application layer), designed for HTTP and HTTPS traffic.
- Offers advanced routing capabilities including path-based and host-based routing.
- Supports routing to multiple target types like EC2 instances, containers (ECS), and Lambda functions.
- Supports WebSockets, HTTP/2, and gRPC.
- Ideal for microservices, container-based architectures, and applications requiring content-based routing.
- Provides performance improvements and enhanced visibility over CLB.

## Network Load Balancer (NLB)

- Operates at Layer 4 (transport layer), handling TCP, UDP, and TLS traffic.
- Designed for ultra-high performance and low latency, capable of handling millions of requests per second.
- Supports static IP addresses and elastic IPs, enabling integration with on-premises environments.
- Suitable for applications requiring extreme performance, static IP addresses, or handling volatile traffic patterns.
- Handles sudden spikes in traffic with minimal latency impact.

***

| Feature | Classic Load Balancer (CLB) | Application Load Balancer (ALB) | Network Load Balancer (NLB) |
|---------|-----------------------------|--------------------------------|-----------------------------|
| OSI Layer | 4 & 7 | 7 | 4 |
| Protocols Supported | HTTP, HTTPS, TCP, SSL | HTTP, HTTPS, WebSocket, HTTP/2 | TCP, UDP, TLS |
| Advanced Routing | No | Yes (path, host-based) | No |
| Target Types | EC2 Instances only | EC2, ECS containers, Lambda | EC2, IP Addresses |
| Performance | Moderate | Moderate | High |
| Static IP Support | No | No | Yes |
| Use Case | Legacy apps, simple load balancing | Microservices, container-based apps | High-performance, low latency |

AWS recommends using ALB for modern application architectures requiring advanced routing and NLB for extreme performance or non-HTTP use cases, while CLB is maintained only for legacy support.[3][4][1]

## 1. Amazon RDS (Relational Database Service)

Amazon RDS is a fully managed relational database service that handles routine database tasks such as provisioning, patching, backup, recovery, and scaling for popular engines including MySQL and PostgreSQL.

- **MySQL / PostgreSQL**: RDS provides managed MySQL and PostgreSQL with support for major versions, automatic minor version upgrades, and compatibility with standard tools and drivers. Amazon Aurora is an enhanced version with higher performance and scalability.
- **Automated Backups**: RDS automatically performs daily backups and stores transaction logs for point-in-time recovery. Backup retention can be set from 0 to 35 days. Backups run without impacting database performance by utilizing storage snapshots.
- **Multi-AZ Deployments**: RDS supports Multi-Availability Zone deployments for disaster recovery and high availability. It synchronously replicates data to a standby instance in a different AZ, failing over automatically if the primary instance becomes unavailable.
- **High availability and failover optimization**: Amazon RDS continuously monitors instance health and automatically restarts or fails over to minimize downtime. Integration with Amazon RDS Proxy can reduce failover time by up to 66%.
- **Monitoring and Metrics**: Enhanced Monitoring captures operating system and database process metrics in real time, integrated with CloudWatch for alerting and visualization.
- **Blue/Green Deployments**: Allows safer database upgrades and schema modifications with zero downtime by creating a staging environment synchronized with production.
- **Migration Support**: Tools and snapshots support fast migration from existing MySQL or PostgreSQL databases to RDS.

***

## 2. Amazon DynamoDB

DynamoDB is a fully managed, serverless NoSQL database built for high availability, fast performance, and seamless scaling.

- **Partitions & Keys**: Data is distributed across partitions using partition keys. Each partition has a maximum throughput limit (3,000 read units/sec and 1,000 write units/sec). Effective partition key design is critical for performance. Optionally, a sort key allows additional data ordering and querying.
- **Capacity Modes**:
  - **Provisioned Mode**: You specify read and write capacity units (RCU/WCU) upfront. DynamoDB provides performance guarantees, with burst capacity and adaptive capacity to handle traffic fluctuations across partitions.
  - **On-Demand Mode**: Automatically scales capacity up or down based on traffic with no capacity planning required, suitable for unpredictable workloads.
- **Performance Optimization**: Adaptive capacity redistributes unused throughput to hot partitions dynamically. Token bucket mechanisms manage traffic flow per partition ensuring consistent performance.
- **Strongly Consistent vs Eventually Consistent Reads**: DynamoDB offers the choice of consistency models for reads with varying latency and availability trade-offs.

***

## 3. Amazon ElastiCache

ElastiCache provides managed in-memory caching services to improve application performance by reducing latency on data retrieval.

- **Redis**:
  - Offers rich data types (strings, hashes, lists, sets, sorted sets) for complex caching needs.
  - Supports data persistence (snapshots), replication, and high availability via Redis Cluster with failover.
  - Advanced features include pub/sub messaging, Lua scripting, transactions, and geospatial support.
  - Single-threaded architecture but supports clustering for horizontal scaling.
- **Memcached**:
  - Simple, multi-threaded caching service optimized for large-scale, distributed caching.
  - Ideal for read-heavy and less complex caching scenarios without persistence or replication.
  - Lacks built-in support for advanced data types or durability features.

***

## 4. Amazon EFS (Elastic File System)

Amazon EFS provides scalable, highly available, and fully managed elastic file storage accessible concurrently by multiple EC2 instances, containers, and on-premises servers.

- **Scalability**: Automatically scales storage to petabytes without manual provisioning.
- **Performance Modes**: Offers General Purpose mode optimized for latency-sensitive workloads and Max I/O mode optimized for high throughput parallel access.
- **Storage Classes**:
  - **EFS Standard**: For frequently accessed files.
  - **EFS Infrequent Access (IA)**: Cost-optimized for files accessed less frequently.
  - **EFS Archive**: Lowest-cost tier for long-term retention with hours-long retrieval.
- **Lifecycle Management**: Automatically transitions files between storage classes based on access patterns to optimize costs.
- **Access Protocols**: Supports NFS v4.0 and v4.1, enabling shared file access with file locking and traditional file permissions.
- **Multi-AZ Availability**: File systems replicate across multiple AZs within a region to ensure data durability and availability.
- **Integration**: Works seamlessly with AWS compute services including EC2, ECS, and Lambda.

***

## 5. AWS Storage Gateway

Storage Gateway is a hybrid storage service bridging on-premises environments with AWS cloud storage for backup, archiving, and disaster recovery.

- **Gateway Types**:
  - **File Gateway**: Presents NFS or SMB file interfaces; stores files as native S3 objects. Ideal for file sharing and backup scenarios needing scalable cloud storage with local caching for low-latency access.
  - **Volume Gateway**: Provides block storage using iSCSI protocol with two modes:
    - **Cached Mode**: Keeps frequently accessed data locally and stores full data in Amazon S3.
    - **Stored Mode**: Retains entire dataset on-premises with asynchronous cloud backup.
  - **Tape Gateway**: Emulates virtual tape libraries (VTL) that integrate with existing backup software; stores backup data in Amazon S3 and archives to Glacier.
  - **Amazon FSx File Gateway**: Supports fully managed file storage for Windows-based environments via Amazon FSx.
- **Protocols Supported**: NFS, SMB (File Gateway), iSCSI (Volume Gateway), and VTL for Tape Gateway.
- **Use Cases**: Seamlessly integrate cloud storage into existing workflows with minimal application changes, achieve backup/recovery goals, and support hybrid cloud elasticity.

## AWS Lambda

AWS Lambda is a serverless compute service that runs your code in response to events without provisioning or managing servers.

- **Triggers**: Lambda functions can be triggered by AWS services such as S3 (object uploads), DynamoDB (streams), API Gateway (HTTP requests), SNS (notifications), CloudWatch Events, EventBridge, and more.
- **Memory & Timeout**: Memory allocation ranges from 128 MB to 10,240 MB, directly affecting CPU and network throughput. The maximum timeout per function invocation is 15 minutes.
- **Common Use Cases**: Real-time file processing, backend for web/mobile apps, ETL workflows, data transformations, event-driven microservices, serverless APIs, and automation tasks.
- **Scaling**: Lambda automatically manages scaling, running multiple instances in parallel to match the incoming request rate.
- **Monitoring**: Integrated with CloudWatch Logs and AWS X-Ray for tracing and performance insights.

***

## Amazon API Gateway

Amazon API Gateway enables developers to create, publish, secure, and monitor APIs at any scale.

- **Types of APIs**: Supports REST APIs, HTTP APIs (simpler and cost-effective), and WebSocket APIs for real-time two-way communication.
- **Security**: Integrates with IAM, Cognito, and Lambda authorizers for flexible authentication and authorization.
- **Traffic Management**: Supports throttling, caching, and request validation to optimize performance and control access.
- **Monitoring & Logging**: Supports CloudWatch metrics and logging, along with X-Ray tracing.

***

## Amazon SNS (Simple Notification Service)

Amazon SNS is a fully managed pub/sub messaging service for decoupling microservices and sending notifications.

- **Use Cases**: Fan-out pub/sub messaging, time-critical notifications via SMS, email, mobile push, and HTTP endpoints, system alerts.
- **Security**: Supports encryption via AWS KMS and fine-grained access control via resource policies.
- **Delivery Mechanisms**: Push-based delivery to multiple subscribers, supporting multiple protocols.

***

## Amazon SQS (Simple Queue Service)

Amazon SQS is a fully managed message queuing service for decoupling and scaling microservices, distributed systems, and serverless applications.

- **Features**: Offers two queue types‚ÄîStandard (high throughput, best-effort ordering) and FIFO (exact order, limited throughput).
- **Durability**: Messages are redundantly stored across multiple servers and AZs.
- **Message Locking**: Locks messages while being processed to prevent multiple consumers from processing the same message simultaneously.
- **Security**: Supports server-side encryption, access policies, and VPC endpoints.

***

## Amazon EventBridge

Amazon EventBridge is a serverless event bus that connects application data from your own apps, integrated SaaS applications, and AWS services.

- **Features**: Supports over 200 integrated event sources, event filtering, schema registry, event replay, and transformation.
- **Reliability**: Guarantees at-least-once event delivery, storing events durably across multiple AZs with automatic retries and a 99.99% SLA.
- **Use Cases**: Building event-driven architectures, integrating SaaS platforms, orchestrating serverless workflows.

***

## 1. Amazon ECS (Elastic Container Service) and Amazon EKS (Elastic Kubernetes Service)

- **Amazon ECS**: A fully managed container orchestration service that integrates tightly with AWS infrastructure and services. Uses task definitions to specify containers, supports EC2 and serverless Fargate launch types, manages service discovery, auto-scaling, and load balancing. Ideal for simple, AWS-focused container deployments.
- **Amazon EKS**: Fully managed Kubernetes service that runs the Kubernetes control plane across multiple AZs for high availability. Allows running standard Kubernetes workloads with AWS integrations for identity, networking, and storage. Preferred for teams familiar with Kubernetes and requiring multi-cloud or hybrid deployments.
- **Key Differences**: ECS is simpler to use with deep AWS integration; EKS is Kubernetes-native, supporting Kubernetes APIs and tools. ECS scales automatically, while EKS requires managing autoscaling via Kubernetes constructs.
- **Security**: ECS supports IAM roles per task; EKS uses Kubernetes RBAC and network policies.
- **Use Cases**: ECS for AWS-centric microservices; EKS for complex, multi-cloud strategies or Kubernetes standardization.

## 1. How to Host a Static Website on Amazon S3

- **Setup**: Create an S3 bucket with a globally unique name, enable "Static website hosting" in bucket properties.
- **Upload Content**: Upload HTML, CSS, JavaScript, and media files. Set appropriate permissions to allow public read access or use CloudFront and signed URLs for controlled access.
- **Index and Error Documents**: Specify index.html as the default landing page and an error document for 404s.
- **DNS Setup**: Use Amazon Route 53 to point your domain (via an alias record) to the S3 website endpoint.
- **Security**: Optionally configure bucket policies to restrict access or enable HTTPS through CloudFront.
- **Benefits**: Cost-effective, highly scalable, low maintenance for simple websites with fast global delivery when combined with CDN.

***

## 2. Deploying a Python Web App on EC2

- **Provision EC2**: Launch an EC2 instance with an appropriate AMI (Amazon Linux 2, Ubuntu).
- **Security Groups**: Configure security groups to allow HTTP/HTTPS (ports 80/443) and SSH (port 22) access.
- **Environment Setup**: Install Python, virtual environment tools, and dependencies (Flask/Django or other frameworks).
- **Deploy App**: Upload application code via SSH, Git, or CodeDeploy.
- **Web Server**: Configure a web server like Nginx or Apache as a reverse proxy connected to the application server (Gunicorn/uwsgi).
- **Database**: Connect to Amazon RDS or any other DB service.
- **Elastic IP**: Allocate a static IP or use an Elastic Load Balancer for scalability.
- **Monitoring & Backup**: Enable CloudWatch monitoring, create AMIs/snapshots for recovery.

***

## 3. Serverless API Using Lambda + API Gateway

- **API Gateway Creation**: Define REST or HTTP API with methods and resources.
- **Lambda Functions**: Create Lambda functions that handle requests and generate responses.
- **Integration**: Connect API Gateway methods to Lambda via proxy integration.
- **Authorization**: Secure API with IAM, Cognito, Lambda authorizers, or API keys.
- **Deployment**: Deploy stages (e.g., dev, prod) with custom domain.
- **Monitoring**: Use CloudWatch Logs and metrics for insights on requests and errors.
- **Benefits**: No server management, automatic scaling, pay per request.

***

## 4. Image Upload Pipeline: S3 + Lambda + DynamoDB

- **Upload to S3**: Client uploads images to a pre-signed S3 URL or directly with appropriate permissions.
- **S3 Event Trigger**: S3 triggers a Lambda function on object creation events.
- **Lambda Processing**: The Lambda function validates, resizes, thumbnails, or processes images as needed.
- **Metadata Storage**: Lambda writes image metadata (filename, upload timestamp, user ID) into DynamoDB for indexing and quick lookup.
- **Security**: Use IAM roles to restrict Lambda and S3 actions; validate inputs in Lambda.
- **Use Cases**: Media hosting platforms, user-generated content apps.

***

## 5. Real-Time Notifications with SNS and SQS

- **SNS Topics**: Create SNS topics for different notification channels.
- **Subscribers**: Add subscribers such as email, SMS, Lambda, or SQS queues.
- **Publishing Messages**: Publishers send messages to SNS topics.
- **SQS Queue**: For reliable delivery and buffering, subscribe SQS queues to SNS.
- **Processing**: Consumers poll SQS, process notifications asynchronously.
- **Use Cases**: Alerts, workflow triggers, decoupling microservices.

***

## 6. VPC Design Examples: Public and Private Subnets

- **Public Subnets**: Subnets with a route to an Internet Gateway, hosting resources that must be reachable from the internet like web servers.
- **Private Subnets**: Subnets without direct internet access; resources like databases and app servers reside here for enhanced security.
- **NAT Gateway**: Placed in public subnet to enable outbound internet access from private subnets (for updates, API calls).
- **Route Tables**: Separate route tables for public and private subnets define traffic routing.
- **Security Groups & NACLs**: Fine-grained security controls at instance and subnet level.
- **High Availability**: Deploy subnets across multiple AZs to build fault-tolerant architectures.
- **Use Cases**: Multi-tier web applications, security-compliant network segregation.
